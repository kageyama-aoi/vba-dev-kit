# VBA開発プロジェクト ナレッジ制御ファイル

このファイルはClaudeが自動で読み込む指示ファイルです。
以下のルールに従い、各ナレッジMDを適切なタイミングで参照してください。

---

## ナレッジファイルの参照ルール

### 常時参照（VBAコードを書くすべての場面）

**`docs/VBA注意事項.md` を必ず読んでから実装すること。**

- VBAコードを新規作成するとき
- 既存コードを修正・追加するとき
- バグを修正するとき

このファイルには「やらかしやすい実装ミス」が蓄積されています。
読まずに実装すると、文字コードエラー・ByRefエラー・パフォーマンス問題が再発します。

---

### 場面別参照

| 状況 | 参照ファイル | タイミング |
|------|------------|-----------|
| **新しいVBAモジュールを作る** | `docs/VBAツール標準テンプレート.md` | コードを書く前 |
| **関数・変数・定数に名前をつける** | `docs/関数命名ルール.md` | 命名するたびに確認 |
| **ユーザーの要件が曖昧・雑多** | `docs/雑多指示を整えるプロンプト.md` | 実装前に要件を整理してから着手 |

---

## 実装着手前の必須ルール

以下を**必ず守ること**。推論で勝手に実装に進まないこと。

1. **PLAN.md や .spec フォルダを読んだ後は、次のアクションをユーザーに確認してから着手すること。**
   - ファイルを参照しただけで「直すべき」と推論して実装に進まない
   - 「修正しますか？」と一言確認する

2. **実装に着手すると判断した時点で、最初に `github-issue-dev` スキルを呼ぶこと。**
   - ユーザーが「直して」と明示した場合も、推論で着手する場合も同様
   - Issue 登録 → 実装 → コミットの順を守る

---

## 開発サイクル（高速化のための標準手順）

```
1. 要件が曖昧なら → docs/雑多指示を整えるプロンプト.md で構造化
        ↓
2. docs/VBA注意事項.md を読む（毎回）
        ↓
3. 新規モジュールなら → docs/VBAツール標準テンプレート.md でコード骨格を作る
        ↓
4. docs/関数命名ルール.md を確認しながら実装
        ↓
5. 実装完了 → コードレビュー（注意事項MDの各項目を満たしているか確認）
```

---

## コード品質の最低基準

以下をすべて満たしていることを確認してから実装完了とすること。

- [ ] `Option Explicit` が先頭にある
- [ ] モジュール変数は `Private` で宣言されている
- [ ] ループ内でセルを直接操作していない（配列経由で処理）
- [ ] ファイル読み込みは ADODB.Stream（UTF-8対応）
- [ ] エラー時に `ScreenUpdating / Calculation / EnableEvents` が必ず復元される
- [ ] 文字列リテラルの繰り返しは `Private Const` に抽出されている
- [ ] 関数名は「動詞 + 名詞」になっている

## スキル一覧（.agent/skills/）

| スキル名 | 使うタイミング |
|---------|--------------|
| `vba-spec` | 「〇〇したい」という雑多な依頼を受けたとき・最初に実行 |
| `vba-new-module` | 新しい .cls / .bas を作るとき |
| `vba-review` | 実装完了後・修正後のレビュー |
| `vba-implement` | 仕様から .cls ファイルまで一気通貫で作るとき |
| `vba-knowledge-update` | 新しい知見をナレッジに追記・並び替えするとき |
| `make_project` | 新規プロジェクトを立ち上げるとき |

## ナレッジの更新ルール

開発中に新しい注意事項・パターンが見つかったら、
そのつど対応するMDファイルに追記してください。

| 発見した知見の種類 | 追記先 |
|----------------|-------|
| VBA特有のエラー・制約 | `docs/VBA注意事項.md` |
| コード構造のベストプラクティス | `docs/VBAツール標準テンプレート.md` |
| 命名に関するルール | `docs/関数命名ルール.md` |

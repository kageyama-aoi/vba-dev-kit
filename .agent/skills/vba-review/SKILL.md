# vba-review｜VBAコードレビューワークフロー

レビュー対象の .cls / .bas ファイルを受け取ったとき、このファイルの手順に従うこと。

---

## ステップ1：対象ファイルを読む

ユーザーが指定したファイルを読み込む。
指定がない場合はプロジェクト内の .cls / .bas ファイルを一覧して確認する。

---

## ステップ2：注意事項チェック（`docs/VBA注意事項.md` 対照）

**`docs/VBA注意事項.md` を読み込み、目次（A〜D）の全項目をチェックリストとして使う。**
チェック項目をこのファイルに書き写さないこと（項目はナレッジ側で増減するため、常に最新の目次を正とする）。

セクションごとの適用条件：

| セクション | 適用条件 |
|-----------|---------|
| A：モジュール骨格 | 常にチェック |
| B：Excel操作パターン | セル・シート・図形の操作がある場合のみ |
| C：コーディング落とし穴 | コード内に該当処理がある場合のみ |
| D：外部ブック操作マクロ | マクロブック自身とは別のブックを操作対象とする構成の場合のみ（コードを読んで判断し、判断理由を報告に含める） |

各項目を ✓/✗ と該当箇所（行番号）で記録する。

---

## ステップ3：品質最低基準チェック

| 項目 | 結果 |
|------|------|
| `Option Explicit` が先頭にある | ✓/✗ |
| ループ内でセルを直接操作していない | ✓/✗ |
| エラー時に ScreenUpdating 等が復元される | ✓/✗ |
| 文字列リテラルの繰り返しは Private Const にある | ✓/✗ |
| 関数名は「動詞 + 名詞」になっている | ✓/✗ |

---

## ステップ4：結果を以下の形式で報告する

```
## VBA チェックレビュー結果

### 適用チェック
- A（骨格）／B（Excel操作）／C（落とし穴）：[適用したセクションと項目数]
- D（外部ブック操作）：[適用 / 非適用 ＋ 判断理由1行]

### 🔴 要修正
- **X-n：項目名**
  問題のある箇所（行番号があれば記載）と、何がまずいかを1〜2行で説明。修正案をセットで提示

### 🟡 要確認
- **X-n：項目名**
  文脈によっては問題になるパターン、または注意が必要な箇所

### ✅ 問題なし
A: 1, 3, 5 … ／ B: … ／ C: … ／ D: …（番号だけ列挙。非適用セクションは「非適用」）

---
**総評：** X件の要修正、Y件の要確認
```

- 問題がない項目は番号だけ列挙して詳細は省く。問題がある項目だけ詳しく説明する
- すべて ✓ の場合は「レビュー完了：問題なし」と報告する
- 修正するかどうかはユーザーに確認してから行う
- レビュー中に新しい落とし穴を発見したら、vba-knowledge-update スキルで `docs/VBA注意事項.md` への追記を提案する

# Git 履歴書き換え手順書

過去のコミットに含まれた機密情報（会社名・個人名・パスワードなど）を
GitHub の履歴からも完全に削除する手順です。

---

## いつ使うか

- コミット済みのファイルに会社名・顧客名・パスワードなどが含まれていた
- `git push` 後に気づいた場合も、この手順で履歴ごと上書きできる

---

## 前提・注意事項

| 項目 | 内容 |
|---|---|
| **影響範囲** | リモートの履歴が書き換わる（force push） |
| **他の人への影響** | clone・pull していた人は `git pull` できなくなる（個人リポジトリなら基本問題なし） |
| **作業前にすること** | 現在の状態を確認し、残したいファイルの変更が作業ディレクトリにあることを確認する |

---

## 手順

### 1. 現在の履歴を確認する

```bash
git log --oneline
```

例：
```
ef738d2  fix: ファイル名修正        ← 最新（情報は除去済み）
c874b21  feat: 初期構成を追加       ← ここに機密情報が含まれていた
121d6c4  Initial commit             ← 残したい起点のコミット
```

### 2. 残したい起点のコミットまで巻き戻す

`--soft` オプションを使うと、**ファイルの内容は現状維持**のまま
コミット履歴だけを指定地点まで戻せる。

```bash
git reset --soft <残したいコミットのハッシュ>
```

例：
```bash
git reset --soft 121d6c4
```

> `git status` で `Changes to be committed:` にファイルが並んでいれば OK

### 3. ファイルの内容を確認・修正する

機密情報がまだファイルに残っている場合はここで修正する。
（すでに修正済みであればそのまま次へ）

```bash
# 修正が必要なファイルを編集後、再ステージ
git add <ファイル名>
```

### 4. 綺麗なコミットを作り直す

```bash
git commit -m "feat: 初期構成を追加（機密情報なし）"
```

### 5. force push でリモートの履歴を上書きする

```bash
git push --force origin main
```

### 6. GitHub 上で履歴を確認する

```bash
git log --oneline
# または GitHub のページで Commits を確認
```

機密情報が含まれていたコミットが消えていれば完了。

## よくある質問

**Q. `--force` と `--force-with-lease` の違いは？**
`--force-with-lease` は「自分が知らない他人のコミットがある場合は上書きしない」より安全なオプション。
チームリポジトリでは `--force-with-lease` を推奨。

```bash
git push --force-with-lease origin main
```

**Q. 複数ブランチに同じ情報が含まれていた場合は？**
ブランチごとに同じ手順を繰り返す。または `git filter-repo` を使うと一括対応できる。

**Q. 該当コミットが途中にあって起点まで戻せない場合は？**
`git rebase -i` で特定コミットだけを編集する方法もある（別途手順書参照）。

## 参考コマンド早見表

| コマンド | 説明 |
|---|---|
| `git log --oneline` | 履歴一覧を確認 |
| `git reset --soft <hash>` | 指定コミットまで戻す（ファイル変更は保持） |
| `git reset --mixed <hash>` | 指定コミットまで戻す（ステージもリセット） |
| `git reset --hard <hash>` | 指定コミットまで戻す（ファイル変更も破棄） ⚠️ |
| `git push --force origin main` | リモートを強制上書き |
| `git push --force-with-lease origin main` | より安全な強制上書き |
